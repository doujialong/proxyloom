> 审计结果：三轮完成；确定性问题已修复，后续二次审计的 D-017 至 D-021 也已确认并写入规范。

## 1. 审计方法

本次审计分三轮进行，每轮采用不同关注面，避免只做措辞检查：

1. 产品范围、术语、需求一致性和边界。
2. 技术可行性、安全性、健康检查和数据生命周期。
3. 可验收性、优先级、用户体验、发布门槛和迭代闭环。

严重级别用于表示问题若未修复对实现的影响：Critical 会造成敏感数据或核心结果风险；High 会造成架构返工、数据错误或核心流程不可用；Medium 会造成行为歧义、运维或验收困难；Low 为局部一致性和发布质量问题。

## 5. 剩余风险

- D-004、D-015 和 D-016 已确认；实现仍需通过许可证清单和进程边界测试验证组合分发方式。
- “所有协议”是持续兼容承诺；未公开或无法合法互操作的私有协议只能保留原始数据并公开阻塞原因，不能承诺虚假转换。
- D-002 已确认密钥放在数据卷之外；部署文档必须防止用户为了方便又把两者放回同一卷。
- 真实节点健康状态仍受本机网络、DNS、探测端点和上游限流影响，D-003、D-007、D-011 的组合行为需要端到端故障注入验证。
- 二次复审已发现 NodeLoom 存在多个精确同名 GitHub 仓库及命名空间，D-020 已确认使用 ProxyLoom，旧工作名禁止用于正式发布。

## 6. 审计完成标准

- 所有确定性 Critical/High/Medium 问题已修入需求正文。
- D-001 至 D-021 均有确认记录、落地阶段和正文中的规范性行为。
- 需求编号和决策引用无重复、无悬空引用。
- P0、P1、P2 与“持续”均有发布或兼容性门槛。
- 核心数据、安全、健康、构建、API、UI、运维和 GitHub 发布均有可验收要求。
