# ProxyLoom 自托管代理订阅管理器需求文档

> 项目名称：ProxyLoom
>
> 文档状态：M0 Baseline v0.8（D-001 至 D-021 已确认，M0 独立三轮复审完成）
>
> 更新日期：2026-07-18
>
> 首要使用场景：Momo v1.2.1 + sing-box v1.12.25，自托管单容器部署

## 1. 背景

现有订阅转换工具通常会先把输入解析成自身的通用节点模型，再渲染为目标格式。这种方式便于跨格式转换，但遇到新协议、新字段或客户端私有扩展时，容易出现字段丢失、值被改写，甚至同格式输入仍无法输出的问题。

本项目从头实现一个面向个人、家庭和小型自托管环境的订阅管理器。它既要具备主流工具的订阅聚合、过滤、重命名、转换、模板和发布能力，也要优先解决以下问题：

1. 同格式输入和输出时支持无损直通，未知字段不得静默丢失。
2. 分别监测订阅源和节点的健康状态，并稳定、及时地过滤不可用项。
3. 多订阅聚合时自动解决节点名称冲突，命名稳定且可追溯。
4. 生成结果可预览、可解释、可回滚，发生错误时不发布损坏配置。
5. 持续覆盖市面上公开、可合法实现的代理协议，新协议出现时无需重做核心架构。

## 2. 产品目标

### 2.1 核心目标

- 通过一个易用的 Web 界面管理订阅源、节点集合、处理流水线、模板和发布端点。
- 以 sing-box 原生 JSON 为第一优先级，完整支持节点列表和完整配置两种输入、输出形态。
- 以市面代理协议持续全覆盖为长期核心目标，而不是只支持少数常见协议。
- 同时保存原始数据和标准化视图，在无损直通、局部修改和跨格式转换之间做明确区分。
- 自动刷新、检查、生成和发布，日常使用不要求手工维护配置文件。
- 单 Docker 服务容器即可部署，业务数据使用一个持久化目录，主密钥通过独立只读 secret 文件挂载。
- 为后续增加 Mihomo、更多协议、通知和安全脚本扩展保留稳定接口。

### 2.2 产品原则

1. **无损优先**：能够保留的数据不得因解析器暂不认识而丢失。
2. **不静默失败**：解析失败、字段降级、节点被过滤和模板引用失效必须可见。
3. **稳定优先**：短暂网络波动不应导致发布内容被清空或大规模抖动。
4. **最小修改**：重命名或覆写节点时，只修改明确要求的字段。
5. **结果可解释**：用户能查看每个节点来自哪里、经历了哪些处理、为何被排除。
6. **默认好用，高级可控**：常见流程使用合理默认值，高级行为允许逐项覆盖。
7. **协议开放**：协议能力由独立适配器扩展，核心模型不得假设协议集合固定不变。

### 2.3 暂不纳入首版

- 公共 SaaS、计费、套餐销售和多人租户隔离。
- 充当长期运行的代理网关或流量转发核心。
- 默认执行来源订阅中携带的任意脚本。
- 首版不要求每种协议同时具备跨格式转换、可视化编辑和真实代理健康检查，但同格式无损直通是底线，其他能力必须按公开矩阵持续补齐。
- 与 Sub-Store 的数据库、私有脚本或接口做到完全兼容。

## 3. 目标用户与典型场景

### 3.1 目标用户

- 在 NAS、家庭服务器或 VPS 上维护多个代理订阅的个人用户。
- 使用 Momo、sing-box 或 Mihomo，需要生成完整客户端配置的高级用户。
- 需要保留 ECH、Reality、uTLS 等较新字段，不接受有损转换的用户。
- 希望自动排除失效订阅和节点，同时保留故障记录的用户。

### 3.2 典型场景

1. 导入一个原生 sing-box 完整配置，提取其中的节点，与其他订阅聚合后填入 Momo 模板。
2. 导入 sing-box 节点并仍输出 sing-box；除发生名称冲突需要修改 `tag` 外，其余字段和值保持不变。
3. 同一服务网站有多条订阅，多个节点都叫“香港 01”；系统保留全部可用节点，并用稳定数字后缀消除冲突。
4. 某订阅暂时返回 500；系统标记订阅异常，但继续使用最后一次成功快照，不立即清空输出。
5. 某节点连续多次无法完成真实代理访问；健康感知输出自动排除它，恢复后再自动加入。
6. 发布前使用目标配置验证器校验完整配置；新生成结果不合法时继续提供上一份有效结果。

## 4. 领域模型

| 概念 | 定义 |
| --- | --- |
| 订阅源 Source | 一个远程 URL、手工粘贴内容或上传文件，包含来源别名、请求参数和刷新策略。 |
| 刷新记录 Refresh Attempt | 一次拉取尝试的请求结果、阶段耗时和错误；失败记录不会覆盖有效快照。 |
| 源快照 Snapshot | 一次通过有效性门槛的原始响应、元数据、解析结果和内容摘要。历史快照可用于对比和回滚。 |
| 原始文档 Raw Document | 完整保存的输入响应或上传内容，用于整份直通、重新解析和审计。 |
| 原始节点 Raw Node | 从源快照中切出的原生节点对象或原始 URI，保留未知字段和原始类型。 |
| 标准化节点 Canonical Node | 用于搜索、健康检查、重复识别、跨格式转换的内部视图；不得取代原始节点。 |
| 节点标识 Fingerprint | 根据协议及关键连接属性生成的带密钥摘要，不以显示名称或列表顺序作为身份，也不暴露认证信息。 |
| 节点出现记录 Node Occurrence | 某来源中一个节点的持久化逻辑记录，用于关联跨快照身份、名称分配、来源和历史状态。 |
| 有效节点 Effective Node | 原始节点应用当前流水线补丁和目标配置后的派生连接对象，不作为原始数据替代品。 |
| 健康键 Health Key | 对有效连接字段、执行器及版本、探测策略和网络上下文计算的带密钥摘要，用于判断健康结果能否安全复用。 |
| 名称分配 Name Allocation | 发布目标内某节点出现记录与最终唯一名称的持久化映射，包含分配版本和保留状态。 |
| 集合 Collection | 一个或多个订阅源及手工节点的逻辑组合。 |
| 流水线 Pipeline | 按顺序执行过滤、重命名、重复识别、排序、属性覆写和分组等操作。 |
| 模板 Template | 用于生成完整 sing-box/Mihomo 配置的基础结构和节点填充规则。 |
| 目标配置 Target Profile | 目标客户端、精确客户端版本、内核类型、精确内核版本及格式能力的组合，例如 Momo v1.2.1 + sing-box v1.12.25 或独立上游 sing-box v1.12.25。 |
| 资源修订 Revision | 来源配置、集合、流水线、模板或发布目标的一次不可变版本，草稿和已发布状态分离。 |
| 发布目标 Output | 将集合、流水线、模板和目标格式组合成稳定订阅地址的配置。 |
| 构建 Artifact | 某次发布目标生成的不可变结果，包含输入版本清单、校验状态、统计、差异和生成日志。 |
| 后台任务 Job | 一次刷新、健康检查、构建、清理、备份或恢复操作，具有稳定 ID、状态、输入修订和重试信息。 |
| 健康记录 Health Record | 订阅源或节点一次检查的结果、耗时、失败类型和时间。 |

节点必须采用“双表示”模型：原始节点负责无损保留，标准化节点负责理解和操作。所有流水线操作记录为补丁；同格式输出时在原始节点上应用最小补丁，而不是从标准化模型重新构造整个节点。

## 5. 功能需求

需求优先级采用：P0 为首个完整可用版本必需，P1 为正式版重要能力，P2 为增强能力；“持续”表示每个兼容性版本都必须履行的长期义务，不作为某个版本可以一次性完成的范围。

### 5.1 订阅源管理

#### SRC-001 多种输入方式（P0）

支持以下输入：

- HTTP/HTTPS 订阅 URL。
- 手工粘贴的节点、节点列表或完整配置。
- 本地上传的 JSON、JSONC、YAML 或文本文件。
- 后续可选的受控本地文件挂载。

验收标准：保存前能够预览检测到的格式、节点数量、协议分布和解析警告。

#### SRC-002 格式检测与手工指定（P0）

- 自动检测当前版本已支持的输入格式；P0 包括 sing-box、Base64 URI 列表和纯 URI 列表，P1 增加 Mihomo/Clash。
- 对能够识别但当前能力矩阵尚未启用的格式，只显示识别结果和所需版本，不得继续执行不完整解析。
- 用户可以覆盖自动检测结果。
- 检测不确定时不得武断转换，应显示候选格式和原因。

#### SRC-003 请求配置（P0）

- 每个源可设置 User-Agent、请求头、超时、重试次数和更新周期。
- 支持 ETag、Last-Modified 和条件请求。
- 支持设置最大响应大小，防止异常内容耗尽资源。
- 全局设置提供不可由普通来源配置突破的硬上限；来源级大小、节点数和解析深度限制只能在硬上限内收紧。
- 订阅 URL、Authorization、Cookie 等敏感信息在界面和日志中默认脱敏。
- 自定义请求头需过滤 hop-by-hop 头；跨主机重定向不得自动转发 Authorization、Cookie 或用户自定义秘密头。
- P1 支持通过指定代理下载订阅。

#### SRC-004 定时刷新与快照（P0）

- 支持手工刷新和按分钟、小时、天配置的周期刷新。
- 调度时间加入少量抖动，避免所有源同时请求。
- 同一资源的刷新任务不得重叠；失败重试采用带抖动的指数退避，并在合理范围内尊重 `Retry-After`。
- 仅成功拉取且通过解析门槛的内容成为新的有效快照。
- 保存最后一次成功快照、最近失败信息和按保留策略管理的历史快照；默认每个来源最多保留 30 个且不超过 30 天。
- 支持查看前后快照的节点新增、删除、改名及协议变化。

#### SRC-005 内容有效性门槛（P0）

每个源可配置：

- 最少节点数。
- 必须存在的协议或字段，例如至少一个 `tls.ech.enabled=true` 的节点。
- 最大节点骤减比例。
- 是否接受空订阅。

未通过门槛的新快照不得替换最后一次成功快照。

#### SRC-006 完整配置导入语义（P0/P1）

导入当前版本支持的完整配置时，用户必须明确选择用途，系统不得凭内容静默猜测；P0 覆盖 sing-box/Momo，P1 增加 Mihomo：

1. **节点源**：只提取代理节点，内置及逻辑出站不进入节点集合。
2. **模板**：保留 DNS、路由、入站、逻辑出站等配置结构，代理节点位置作为填充点。
3. **模板与节点源**：从同一快照建立两个有版本关联的资源。
4. **整份直通（P1）**：不参与聚合或字段操作，逐字节发布原始文档。

导入预览必须展示每种用途会保留和忽略的顶层字段、代理节点数、逻辑出站数及警告。后续改变用途应创建新修订，不修改已有构建的输入语义。

#### SRC-007 订阅元数据（P1）

- 识别常见 `Subscription-Userinfo` 响应头中的上传、下载、总流量和到期时间。
- 识别常见更新周期及服务主页提示，但外部 URL 只作为不可信文本展示，不能自动获取或跳转。
- 在概览和来源详情展示流量余量、到期状态和最后更新时间，并允许为临近到期或流量耗尽配置通知。
- 元数据缺失、格式错误或与上次相比倒退时显示警告，不影响有效节点快照。

### 5.2 sing-box 无损输入与输出

#### RAW-001 原生对象保留（P0）

- sing-box JSON 中每个节点的完整对象必须原样保存，包括当前解析器不认识的字段。
- 字符串、数字、布尔值、数组、对象和 `null` 的类型不得被擅自改变。
- 支持从完整配置的 `outbounds` 中识别节点，也支持节点数组和单节点对象。
- 能区分节点出站与 `direct`、`block`、`selector`、`urltest` 等逻辑出站。

#### RAW-002 同格式无损直通（P0）

当输入、输出属于同一结构化格式族和兼容目标配置，且某节点未经过字段修改操作时：

- 输出节点所有字段和值必须与原始节点语义一致。
- 未知字段、客户端扩展字段以及嵌套字段必须保留。
- 不要求 JSON 空白、缩进和对象键顺序逐字节相同。
- 若用户选择“完整响应直通”且未进行聚合或修改，P1 可提供逐字节返回原始响应的模式。

验收标准：向节点加入解析器未知的多层嵌套字段，经导入和输出后进行结构化比较，结果完全相等。

#### RAW-003 最小补丁输出（P0）

- 自动重名仅允许修改目标格式定义的显示名称字段，例如 sing-box 的 `tag` 或 Mihomo 的 `name`。
- 属性覆写仅修改用户指定的 JSON 路径，未知的相邻字段必须保留。
- 所有补丁需记录操作来源，并能在预览中显示修改前后差异。
- 某项操作无法无损应用时，构建必须报错或要求用户明确允许有损转换，不能静默降级。

#### RAW-004 跨格式转换提示（P0）

- 跨格式输出允许使用标准化节点模型重新渲染。
- 系统必须列出目标格式无法表达的字段。
- 默认将存在关键字段丢失的构建标记为失败；用户可按发布目标明确允许特定降级。
- 协议、认证、TLS、传输和 ECH/Reality 等连接关键字段不得默认忽略。

#### RAW-005 目标配置与版本（P0）

- 输出必须绑定目标客户端/内核和版本，不能只选择笼统的“sing-box JSON”。
- Momo 兼容配置与上游 sing-box 配置使用不同目标配置，即使两者标称相同版本。
- P0 目标配置基线固定为 Momo v1.2.1 + sing-box v1.12.25，以及独立上游 sing-box v1.12.25；sing-box v1.13.14 是首个兼容性版本目标，不扩大 P0 范围。
- 模板和节点分别校验版本兼容性；目标配置变更需要重新构建，不能复用旧校验结果。
- P1 支持调用匹配目标配置的客户端或内核对完整配置执行正式校验。

### 5.3 协议全覆盖与演进

“支持市面上所有协议”定义为：持续覆盖主流代理生态中已有公开规范、公开实现或能够合法互操作的协议，包括新协议、仍在使用的历史协议和客户端扩展协议。对于没有公开规范、没有可用测试样本或受法律条件限制而无法实现的私有协议，系统必须明确标记阻塞原因，不能伪装成已支持。

#### PROTO-001 全协议覆盖范围（持续）

- 覆盖 sing-box 当前稳定版支持的全部代理出站协议，并跟随其稳定版本演进。
- 覆盖 Mihomo/Clash.Meta 当前稳定版支持的全部代理节点协议，并跟随其稳定版本演进。
- 覆盖主流订阅客户端仍在使用的分享 URI、配置对象和历史协议。
- 覆盖同一协议在不同客户端中的常见字段别名、URI 变体和兼容扩展。
- 新协议即使尚未完成标准化适配，只要输入、输出为相同结构化格式，也必须先通过原始节点机制无损直通。
- 目标客户端本身不支持某协议时，不得声称可以转换；应允许选择兼容目标、仅原样输出或明确失败。

#### PROTO-002 协议能力矩阵（P0）

每个协议必须分别公布以下能力，不能用单一“已支持”掩盖差异：

1. **识别**：可以判断协议类型并提取显示所需的安全元数据。
2. **无损直通**：相同格式输出时完整保留原始节点。
3. **标准化解析**：可以解析连接、认证、TLS 和传输等字段。
4. **编辑与转换**：可以安全修改并渲染到目标格式。
5. **真实健康检查**：可以通过实际协议栈完成代理访问探测。
6. **目标客户端校验**：生成结果经过对应客户端或内核验证。

Web 界面和 API 都应能查询协议能力矩阵、适用版本、已知限制和最近验证时间。

#### PROTO-003 可扩展协议注册表（P0）

- 使用稳定的规范协议 ID 和别名注册表，不使用散落在业务代码中的字符串判断。
- 解析器、标准化映射、指纹、渲染器、字段校验和健康检查驱动分别注册，允许分阶段补齐。
- 协议专有字段进入可保留的扩展字段区，不能为了适配最小公共模型而被删除。
- 协议适配器独立版本化，并声明支持的来源格式、目标格式和客户端版本范围。
- 添加新协议不应要求修改订阅、集合、流水线、快照和发布等核心领域逻辑。

#### PROTO-004 新协议跟进机制（P0/持续）

- 发现未知协议时不得导致整份订阅解析失败；系统保存原始节点、标记协议 ID 并显示待适配状态。
- 建立公开的协议兼容清单，记录发现时间、样本、规范、上游支持情况和实现状态。
- 在获得稳定规范或主流上游实现及脱敏测试样本后，于下一个兼容性版本补充解析与输出适配。
- 新协议适配必须加入脱敏 fixture、无损测试、解析测试、渲染测试和目标内核校验，适用时再加入真实健康检查。
- 上游协议字段变更触发自动兼容性测试；失败时保留同格式无损输出，并明确暂停可能有损的跨格式转换。

#### PROTO-005 支持质量门槛（P0）

- 只有通过对应能力测试的维度才能标记为支持。
- 不得把端口可连接当作协议已通过真实健康检查。
- 不得通过丢弃不认识的参数来换取“成功转换”。
- 不得凭协议名称相同就假设不同客户端的字段语义一致。
- 对已宣布支持的协议保留回归样本，后续版本不得无提示降低能力等级。

### 5.4 聚合与处理流水线

#### PIPE-001 多源聚合（P0）

- 一个集合可以包含多个订阅源和手工节点。
- 每个节点保留来源、快照、原始名称和导入时间等追溯信息。
- 用户可调整源顺序，但节点身份不得因顺序变化而改变。

#### PIPE-002 过滤（P0）

支持组合以下条件：

- 名称正则包含/排除。
- 来源、协议和地区。
- 节点健康状态、最近延迟和连续失败次数。
- 是否包含指定字段，例如 ECH、Reality、UDP 支持。
- P1 支持端口、服务器域名/IP、ASN 和自定义标签。

界面必须显示每一步输入、输出和被排除数量，并能解释单个节点被排除的原因。

#### PIPE-003 重命名（P0）

- 支持正则替换、前缀、后缀、模板变量和地区标准化。
- 可用变量至少包括：来源别名、来源序号、来源主机名、协议、地区、原始名称和稳定短标识。
- 修改后仍发生冲突时，必须进入统一的名称冲突处理器。

#### PIPE-004 重复识别与保留（P0）

- 使用稳定指纹识别同一连接节点，不得仅按名称判断重复。指纹必须覆盖影响连接身份的协议、服务器、端口、认证、TLS 和传输字段。
- 指纹使用持久化本地密钥执行 HMAC-SHA-256 或等价带密钥摘要；数据库、日志和 API 不得保存或返回可用于离线猜测密码的裸摘要输入。
- 指纹密钥作为独立数据密钥由主密钥加密，并包含在加密备份中；轮换主密钥不得改变已有节点指纹。
- 指纹算法和规范化版本写入数据记录；升级算法时提供双读迁移，不能导致所有锁定名称和健康历史瞬间失联。
- 已登记协议使用适配器声明的连接身份字段计算指纹。未知协议的结构化对象使用排除格式级显示名称字段后的规范化原始对象计算不透明指纹；URI 只有在格式规范明确声明片段为显示名称时才排除片段，否则使用完整原文。该指纹只能表示原始结构相同，不得宣称协议语义等价。
- 未知 JSON 对象的规范化必须基于无损有序语法树：对象键按 UTF-8 码点排序、数组保持顺序、字符串保持解码后的精确值、数字保留已验证的原始词法；不得经 IEEE-754 浮点转换后再计算指纹。
- 自动流水线不得因指纹相同而合并或删除节点；来自不同来源或同一来源的重复项全部保留，并保留各自来源记录。
- 对名称相同但连接信息不同或完全相同的节点，都通过名称冲突处理器生成唯一名称。
- 重复识别结果必须可预览；只有 Health Key 完全相同的有效节点才能复用健康检查结果。只有用户显式添加删除/过滤操作，或发布目标已确认的健康过滤策略，才允许减少节点。

#### PIPE-005 排序和属性操作（P1）

- 支持按名称、来源、地区、协议、延迟和自定义优先级稳定排序。
- 支持给标准字段和目标格式字段设置、删除或条件覆写属性。
- 支持给节点添加用户标签，标签不必写入最终客户端配置。

#### PIPE-006 扩展操作（P2）

- 提供版本化插件接口或受限脚本操作。
- 脚本默认无网络、文件系统和宿主环境权限，并设置执行时间与内存限制。
- 脚本产生的每项字段变化仍需进入差异和审计记录。

#### PIPE-007 修订、草稿与并发编辑（P0）

- 来源配置、集合、流水线、模板和发布目标的每次保存都创建不可变修订，不能原地改写已参与构建的内容。
- 草稿修订可预览和试构建；只有明确发布的修订参与后续自动构建。
- 管理 API 使用修订号或 ETag 做乐观并发控制，旧版本保存返回冲突和可比较差异，不能覆盖较新的编辑。
- 后台任务启动时固定输入修订；任务执行期间发生的新编辑只影响后续任务。
- 删除被构建或其他资源引用的修订时，默认转为归档；满足保留和引用条件后才能物理清理。

#### PIPE-008 跨快照节点关联（P0）

- 每个导入节点建立 Node Occurrence；刷新生成新快照时先与同一来源的历史出现记录关联，再进入名称和健康处理。
- 关联优先使用相同连接指纹；连接字段变化时，只能在原始名称、协议、来源位置等辅助属性形成唯一候选时继承出现记录。
- 多个候选同样合理时不得武断继承，应创建新的出现记录、保留旧记录为暂时缺席，并在快照差异中标记歧义。
- 暂时缺席的出现记录在 30 天保留期内不得物理删除，以便节点恢复后找回名称分配和历史；超过 30 天后旧记录可清理，节点再次出现时按新节点处理。
- 跨快照关联算法和版本写入构建清单；算法升级不得静默批量改变已有名称分配。

### 5.5 名称冲突处理

#### NAME-001 冲突范围（P0）

- 在每个发布目标内保证最终节点名称唯一。
- 同时检测单个订阅内部、多个订阅之间、多个同主机订阅之间的重名。
- 检测模板内已有逻辑出站名称，避免节点名称与 `selector`、`urltest` 等出站冲突。

#### NAME-002 自动命名策略（P0）

默认策略：

1. 不冲突的名称保持不变。
2. 重名集合中的第一个节点保持原名，其余节点依次追加 ` #2`、` #3` 等稳定数字后缀，例如 `香港 01`、`香港 01 #2`、`香港 01 #3`。
3. 编号基于持久化来源 ID、节点指纹和原始节点出现标识确定，不依赖本次订阅输入顺序；刷新、源排序和容器重启后保持稳定。
4. 若生成的带编号名称仍与原始名称或逻辑出站冲突，继续递增编号直到唯一；目标格式有长度限制时优先截断基础名称并完整保留数字后缀。
5. 完全相同的节点也全部保留并参与相同编号规则。

支持由用户主动切换为来源前缀、来源后缀、主机名或自定义命名模板，但默认始终使用数字后缀。

#### NAME-003 同网站多订阅（P0）

- 按规范化主机名识别来自同一网站的多个订阅，但不得把主机名当作唯一来源身份。
- 新建同主机订阅时自动生成可编辑别名，例如 `example.com #1`、`example.com #2`。
- 来源别名用于追溯、筛选和可选命名模板，不改变默认数字后缀策略。
- 刷新、源顺序变化和容器重启后，相同节点的最终名称必须保持稳定。

#### NAME-004 命名预览（P0）

保存发布目标前显示原名、最终名称、来源和触发改名的原因。用户可以锁定单个节点的最终名称；锁定名称发生冲突时必须明确报错。

#### NAME-005 名称分配生命周期（P0）

- 数字后缀分配按发布目标持久化，不能在每次构建时根据当前列表临时重算。
- 名称分配在健康过滤之前对发布目标的全部候选 Node Occurrence 执行；`unhealthy`、`unchecked`、`unsupported` 或暂时被过滤的节点仍保留或获得分配，健康变化不得成为其他节点改名的原因。
- 已存在节点新增新的重名节点时，已有节点名称不变，新节点获取新的后缀。
- 某节点消失时其他节点不得自动递补或去掉后缀，避免客户端策略选择随刷新变化。
- 节点缺席后名称分配保留 30 天，期间不得递补或复用空号；节点在期限内重新出现时恢复原分配。
- 超过 30 天后，新节点可以使用当前最小空号；旧节点再返回时按新节点分配。
- 用户可以预览并手工执行“压缩编号”，该操作作为发布目标的新修订，展示所有受影响名称并要求确认。

### 5.6 健康检查

订阅源健康与节点健康是两个独立维度。订阅能成功下载不代表节点可用；订阅暂时下载失败也不代表其上次快照中的节点立即失效。

#### HEALTH-000 状态模型（P0）

- 订阅源状态使用 `unknown`、`healthy`、`degraded`、`unhealthy`、`disabled`。
- 节点状态使用 `unchecked`、`checking`、`healthy`、`degraded`、`unhealthy`、`unsupported`、`disabled`。
- “陈旧”是附加属性，表示快照或检查结果超过配置期限，不替代健康状态。
- 端点检查和真实代理检查分别保存结果；节点总体状态由最新有效检查、协议检查能力和阈值确定。
- 没有检查记录的节点为 `unchecked`；当前执行器不支持其协议的节点为 `unsupported`，两者不得显示为健康。
- 状态转换、触发原因和采用的策略版本必须写入健康记录。

#### HEALTH-001 订阅源健康（P0）

每次刷新至少记录：

- DNS、连接、TLS、HTTP 状态和总耗时。
- 响应大小、内容类型、缓存命中情况。
- 解析成功/失败、节点数量、警告数量和内容摘要。
- 与上次有效快照相比的数量变化。
- 失败类型和经过脱敏的错误信息。

来源状态按 HEALTH-000 统一定义。

#### HEALTH-002 最后可用快照（P0）

- 单次刷新失败不得删除已有节点。
- 失败时继续使用最后一次成功快照，并将来源标记为降级。
- 可配置连续失败阈值、最长陈旧时间和超过期限后的处理方式。
- 超过门槛后，健康感知输出可排除该来源；普通输出可选择继续使用陈旧快照并显示警告。
- 从未成功获取的来源不得以空内容参与发布。

#### HEALTH-003 节点分层检查（P0/P1）

节点检查分为以下层级：

1. **端点检查（P0）**：DNS 解析及 TCP 连接；对适用协议记录基础连通性。此结果不能被标记为“代理完全可用”。
2. **代理访问检查（P0，首批协议）**：使用真实协议栈通过该节点访问指定 HTTP/HTTPS 探测地址，验证期望状态码并记录延迟；首批覆盖 Shadowsocks、VMess、VLESS、Trojan、Hysteria 2、TUIC 和 AnyTLS。
3. **扩展能力检查（P1）**：可选 UDP、DNS、流媒体解锁或指定业务 URL；不同检查结果分开记录。

不允许仅凭 `ping` 或服务器端口开放就宣称代理节点健康。暂不支持真实协议检查的节点应标为 `unsupported`，尚未检查的节点标为 `unchecked`。

#### HEALTH-004 调度与资源控制（P0）

- 支持全局默认策略和按来源/发布目标覆盖。
- 可设置检查周期、超时、并发上限、探测 URL 和期望状态码。
- 默认代理探测项使用可编辑的 Cloudflare 与 Google HTTPS 端点，默认直连控制使用 Cloudflare；用户可替换为自建端点，任何单一公共服务都不能作为唯一判断依据。
- 调度加入抖动和并发队列，避免短时间连接风暴。
- 支持手工检查单个节点、某个来源或全部节点。
- 对长期未被任何发布目标使用的节点支持懒检查。
- 健康检查流量、DNS 和连接必须能在容器环境中正确回收，不泄漏后台进程或端口。
- Health Key 从应用流水线连接补丁后的 Effective Node 计算，排除显示名称和来源标签，但覆盖所有影响拨号、认证、TLS、传输、DNS、detour 的字段，以及执行器/版本、探测目标/期望结果和网络配置版本。
- 仅 Health Key 相同且结果未陈旧的节点可以复用检查结果；原始指纹相同但有效连接、执行器或探测策略不同的节点必须分别检查。
- 每个唯一 Health Key 在同一检查批次最多执行一次实际探测，结果关联到全部等效 Node Occurrence，但各节点仍保留独立展示和名称。
- 检查批次由发布目标、健康策略修订、探测配置修订、执行器/版本、网络上下文和最长 5 分钟的调度窗口共同界定；批量失败保护只统计该批次内已完成的唯一 Health Key，不得把不同发布目标或窗口的失败任意合并。一次实际探测可以通过批次项关联到多个批次，但每个批次独立作出保护判断。
- 真实协议执行器必须有并发、CPU、内存、文件描述符、执行时间和子进程数量限制。
- 临时配置使用最小权限创建并及时清理；标准输出、错误输出和进程参数不得泄露节点密钥。
- 健康执行器采用可插拔外部进程驱动：P0 镜像集成 sing-box 驱动，P1 增加 Mihomo 驱动；执行器严格隔离，并随发布物提供对应许可证材料。
- 公共探测请求不得携带节点名称、来源、订阅标识、Cookie、Authorization 或唯一跟踪 ID；使用通用 User-Agent，并在界面说明探测服务会看到代理出口 IP 及预计流量。

#### HEALTH-005 防抖与恢复（P0）

- 使用连续失败阈值进入 `unhealthy`，默认值为 3 次。
- 使用连续成功阈值恢复为 `healthy`，默认值为 2 次。
- `timeout`、认证失败、TLS 失败、协议不支持和探测站点异常分开计数。
- 每批检查先执行本机直连控制探测，并至少支持两个独立探测目标，区分节点故障与检查基础设施故障。
- 同批至少取得 10 个有效样本且节点归因失败率达到 50% 时暂停批量摘除；有效样本只包含成功或可归因到节点的失败，目标故障、执行器崩溃、资源限制、取消和不支持不进入分子或分母。全部指定直连控制探测均失败时无条件暂停并告警。

#### HEALTH-006 健康过滤（P0）

- 发布目标可启用“排除不健康节点”，并选择是否保留 `unchecked`、`unsupported`、`degraded` 或陈旧节点。
- 默认只在达到连续失败阈值后排除节点，不因一次超时抖动输出。
- 因健康状态被排除和重新加入都必须写入构建差异。
- 过滤后节点数低于发布目标安全门槛时，不发布新结果，继续提供最后一份有效构建。

#### HEALTH-007 展示与通知（P1）

- 仪表盘展示订阅可用率、节点健康数、异常来源、最近刷新和最近构建。
- 节点页展示当前状态、最近延迟、最近若干次检查和失败原因。
- P1 首先支持 Webhook，其次支持 Telegram，并提供冷却和聚合，避免重复轰炸；其他渠道通过通知适配器扩展。

#### HEALTH-008 恢复检查与队列压力（P0）

- 调度优先级依次为手工复测、`unhealthy` 恢复检查、新节点首次检查、失败快速复核和常规周期检查；同优先级内按最早到期时间处理。
- 队列长度必须有硬上限；达到上限时合并重复 Health Key 任务并延后低优先级周期任务，不能无限占用内存或静默丢弃手工任务。
- 界面展示待检查唯一 Health Key 数量、最老任务等待时间、计划覆盖率和预计完成时间；排队超过结果有效期时节点标记陈旧。
- 节点标记 `unhealthy` 后依次在 10、20、30 分钟复查，之后每 60 分钟复查；首次恢复成功后 2 分钟再次确认，连续两次成功才恢复为 `healthy`。
- 默认同时探测 16 个唯一 Health Key，可由管理员在 1 至 64 的受保护范围内调整；默认最多运行 4 个执行器子进程、连接同一服务器 IP 4 个、连接同一探测目标 8 个，并在总并发内预留 2 个手工复测槽位。
- 队列硬上限为 100,000 个唯一 Health Key；上述并发值允许管理员调低，超过参考设备能力时系统可根据资源压力自动降低，但不得自动突破上限。
- 全局物理探测队列必须以 Health Key 唯一化，Probe Batch 只通过关联项引用同一个队列项和 Health Record；不得用每批重复行冒充 100,000 个唯一 Health Key 的硬上限。
- 同一服务器 IP 并发硬上限为 16，同一探测目标并发硬上限为 32，执行器子进程硬上限为 4；提高总并发不得绕过这些独立限制。
- 30 分钟是常规检查的期望周期，不是无条件完成保证。系统根据唯一 Health Key 数、最近探测耗时和当前限制持续计算预计覆盖时间；预计无法在结果陈旧期限内覆盖时必须告警并把超期结果标记为陈旧。
- 调整并发只影响后续任务；已运行执行器仍受创建时的资源限制和关闭宽限期约束。

### 5.7 模板、分组与配置生成

#### BUILD-001 输出形态（P0）

sing-box 至少支持：

- 仅节点 `outbounds` 数组。
- 包含 `outbounds` 数组的 JSON 对象，具体外层结构由目标配置声明。
- 基于模板生成的完整配置。
- P1 支持符合条件时的原始完整响应逐字节直通。

#### BUILD-002 模板管理（P0）

- 模板可粘贴、上传或从 URL 获取，并有独立的刷新、快照和校验状态。
- 远程模板支持 GitHub Raw 等 HTTP(S) 地址、加密保存的自定义请求头、手工刷新及 60 秒到 7 天的自动刷新周期；URL 和请求头在列表、日志、审计与错误中必须脱敏。
- 自动刷新按内容 SHA-256 去重；内容未变化不得触发关联发布目标重建，拉取失败、响应超限、解析失败或模板校验失败不得覆盖最近有效内容。
- 新的有效内容生成模板修订后，必须为所有引用该模板的发布目标生成去重构建任务；排队失败必须可恢复，不得留下永久未重建状态。
- 进程重启后根据已保存刷新周期和最近有效修订恢复计划；失败采用带抖动和上限的指数退避，单个坏模板不得阻塞其他模板或发布读取。
- 节点填充位置和逻辑出站不得依赖脆弱的文本替换。
- 合并后检查所有出站引用，禁止发布引用不存在节点或组的配置。
- 支持复用仓库中 Momo/sing-box 配置所需的完整 DNS、route、inbounds 和 experimental 结构。
- 模板修改遵循 PIPE-007 的草稿、修订和并发控制规则；远程模板的新快照不得静默替换正在发布的已确认修订。

#### BUILD-003 自动分组（P0）

- 根据名称、来源元数据和可选 GeoIP 信息识别香港、日本、新加坡、美国、台湾、韩国、澳洲及其他地区。
- 支持手动选择组、地区自动测速组、全局组和自定义业务组。
- 支持按字段生成特殊组，例如所有 ECH 节点组。
- 组为空时的行为可配置为省略、保留或构建失败。

#### BUILD-004 发布前校验（P0）

- 结构校验：JSON/YAML 可解析、名称唯一、引用完整、关键字段存在。
- 语义校验：协议必要字段和目标版本兼容性。
- P0 每个目标配置都必须提供可自动执行的验证器；验证器可以由内部 schema/语义规则、兼容性 fixture 和可分发的目标工具组合，但必须声明验证强度。
- P1 在能够合法、稳定分发或安装目标版本客户端/内核时，将其真实配置检查纳入发布矩阵。
- 校验失败时不得覆盖最后一份有效构建。

#### BUILD-005 稳定发布地址（P0）

- 每个发布目标提供固定订阅 URL，并由高熵只读令牌授权；URL 是否容易猜测不得作为安全控制。
- 支持独立轮换访问令牌，旧令牌可立即吊销。
- 只读令牌仅保存带密钥摘要；管理界面不重复展示令牌明文，只能轮换。
- 支持正确的 Content-Type、ETag、Last-Modified 和条件响应；默认发送 `Cache-Control: private, no-cache`、`Referrer-Policy: no-referrer` 和 `X-Content-Type-Options: nosniff`。
- 客户端访问默认返回最近有效构建，不在请求线程内同步执行高成本全量刷新。
- 强制刷新接口仅允许经过认证的管理端调用。

#### BUILD-006 构建历史与回滚（P0）

- 按保留策略保存构建产物、构建统计和前后差异；每个输出默认保留 30 个构建，当前发布版本及其直接输入不得被自动清理。
- 用户可以将某个历史构建设为当前发布版本。
- 自动构建恢复正常后，可按策略退出手工锁定状态。

#### BUILD-007 可复现构建（P0）

- 每个构建保存机器可读清单，至少包含输出修订、源快照 ID/内容摘要、集合和流水线修订、模板修订/快照、目标配置及适配器版本、名称分配快照、精确 Health Record ID 与结果、采用的 Probe Batch ID/结论及 Control Probe Record ID、健康策略修订、协议注册表版本、跨快照关联算法版本、服务版本、显式动态输入和构建原因。
- 渲染、排序、重复识别和数字后缀分配必须确定性执行；同一构建清单在同一平台重新执行应产生逐字节相同的产物。
- 模板若需要生成时间等动态值，动态值必须作为已记录的显式构建输入，不能在重放时读取当前时间。
- 回滚切换的是完整构建产物和清单，不重新使用当前来源或当前健康状态计算历史结果。

#### BUILD-008 构建触发与安全门槛（P0）

- 接受新的来源或模板快照、发布新的资源修订、健康状态跨越过滤边界、目标配置更新时触发受影响发布目标的构建。
- 短时间内同一发布目标的多次触发应合并，正在运行的旧输入构建不得覆盖更新输入已成功完成的构建。
- 手工构建返回后台任务 ID；重复提交相同幂等键不得创建重复任务。
- 节点数量安全门槛同时支持绝对最小值和相对降幅；首次构建仅应用绝对门槛。
- 自动刷新或健康变化触发的构建越过门槛时暂停发布；用户主动编辑导致的预期大幅删减可在预览中明确确认后发布。

### 5.8 Web 管理界面

#### UI-001 信息架构（P0）

主导航至少包含：概览、订阅源、节点、集合与流水线、模板、发布目标、运行记录和设置。

#### UI-002 核心工作流（P0）

- 新建订阅源：输入 URL/内容 -> 检测与预览 -> 设置别名和刷新 -> 保存。
- 新建发布目标：选择来源 -> 配置处理 -> 预览节点与改名 -> 选择模板和格式 -> 校验 -> 发布。
- 故障处理：从概览进入异常来源/节点 -> 查看检查历史 -> 手工复测或临时禁用。
- 更新操作尽量不离开当前上下文，批量选择、搜索和筛选必须可用。

#### UI-003 预览与差异（P0）

- 节点预览支持按来源、协议、地区、健康状态和改名状态筛选。
- 构建预览展示节点数量变化、字段降级、重名处理、健康过滤和模板引用问题。
- 原生节点可查看结构化差异；敏感字段默认遮罩，需要明确操作才显示。

#### UI-004 核心易用性（P0）

- 中文为首个完整语言，所有 P0 工作流不得依赖未翻译的内部字段名才能完成。
- 在 360 px 手机宽度到常见桌面宽度下不出现控件遮挡、不可访问操作或横向溢出；大表格可采用受控横向滚动或切换为摘要列表。
- 所有异步操作提供明确的加载、空、部分成功、错误、禁用和重试状态，后台任务可离开页面后继续查看。
- 危险操作提供影响范围说明与二次确认。
- 状态不能仅依赖颜色表达；协议不支持、未检查、检查失败和来源陈旧使用不同文字状态。
- 查看或导出未遮罩秘密需要近期重新认证，并写入审计记录。

#### UI-005 首次使用与恢复（P0）

- 首次启动向导完成管理员、密钥状态、网络访问策略和首个来源初始化，并允许安全跳过非必要步骤。
- 没有来源时直接提供导入入口；导入后可从同一流程创建集合、选择推荐模板、预览并发布首个端点。
- 系统处于主密钥错误、数据库需迁移、构建暂停或探测基础设施故障时，概览显示单一明确主状态和可执行恢复入口。
- 备份恢复先校验文件、版本、密钥和影响范围，恢复前自动备份当前状态。

#### UI-006 可访问性与效率（P1）

- 支持亮色/暗色主题，文本结构为后续国际化留接口。
- 常用导航、搜索、选择和批量操作支持键盘，焦点顺序清晰。
- 基础交互满足 WCAG 2.1 AA 对比度、焦点可见性和表单标签要求。
- 常用筛选条件可以保存为个人视图，大列表返回上一页时保留筛选、排序和滚动位置。

### 5.9 API 与自动化

#### API-001 REST API 契约（P0）

- 管理 API 从 `/api/v1` 起步并提供与实现同步的 OpenAPI 文档；破坏性变更使用新主版本路径。
- JSON 错误响应至少包含稳定错误码、用户可读消息、字段详情和关联 ID，不能要求客户端解析日志文本。
- 时间统一使用带时区的 RFC 3339，ID 不复用；枚举值和分页规则写入 API 文档。
- 列表接口提供有上限的分页、稳定排序和服务端筛选，不能默认一次返回全部节点或健康记录。

#### API-002 并发与异步任务（P0）

- 修改资源时携带修订号或 `If-Match`；版本冲突返回 HTTP 409/412 及当前修订信息。
- 刷新、检查、构建、备份和恢复等长任务返回 HTTP 202 和后台任务 ID，可查询进度、结果和脱敏错误。
- 支持 `Idempotency-Key` 的动作必须定义键有效期和请求体冲突行为。
- 客户端断开连接不得取消已经接受的后台任务；具有破坏性的任务需提供显式取消规则。

#### API-003 导入导出与访问令牌（P0/P1）

- P0 提供版本化的配置导出和预检查导入，导入失败不得留下部分资源。
- P0 发布令牌只能读取绑定的发布目标，不能调用管理 API。
- P1 提供具备最小权限范围和到期时间的管理 API 令牌，创建时只展示一次明文。
- 所有令牌的创建、使用失败、轮换和吊销进入安全审计。

### 5.10 管理、安全与运维

#### OPS-001 单容器部署（P0）

- 提供 Dockerfile 和 Compose 示例。
- 单个数据目录包含加密数据库、构建产物和非秘密配置；主密钥不属于数据目录，通过独立只读 secret 文件挂载。
- 支持 `linux/amd64` 和 `linux/arm64`。
- 容器提供健康检查，支持设置时区、监听地址和端口。
- 正式镜像以非 root 用户运行；根文件系统应可配置为只读。
- 只读根文件系统模式为执行器提供带大小上限、`noexec`、`nosuid` 的 tmpfs 临时目录；除数据目录和该临时目录外不要求其他可写路径。

#### OPS-002 数据存储（P0）

- 使用 SQLite，提供版本化迁移和启动前兼容性检查。
- SQLite 数据目录必须位于受支持的本地文件系统；不承诺在 NFS/SMB 数据库文件上可靠运行。
- 写入构建和切换当前发布版本必须是原子操作。
- 迁移前自动完成可验证备份，迁移失败不得用半迁移数据库继续启动。
- 提供一致性备份与恢复，导出内容包含模式版本、校验摘要和恢复说明。
- 自动化测试必须覆盖从受支持的旧版本升级、备份恢复和主密钥错误三种路径。

#### SEC-000 威胁模型与安全基线（P0）

- 订阅源、模板、节点字段、分享链接、HTTP 响应和上传文件全部视为不可信输入。
- 订阅 URL、请求秘密、节点凭据、私钥、原始快照、构建产物、发布令牌和含密钥备份属于敏感数据。
- 默认威胁包括恶意订阅内容、恶意模板、同局域网未授权访问、浏览器跨站请求、服务端请求伪造、凭据日志泄露和数据卷离线复制。
- 默认不发送遥测；未来增加遥测时必须为明确选择加入，且不得包含订阅、节点或发布端点数据。
- 管理端直接暴露到非可信网络时必须使用 HTTPS；服务可依赖受信反向代理终止 TLS，但必须文档化安全配置。

#### SEC-001 管理端认证（P0）

- 首次启动完成管理员初始化。
- 管理 API 使用会话或令牌认证；发布端点使用独立只读令牌。
- 管理员密码使用 Argon2id 或等价的内存困难算法保存，不得使用可逆加密或快速裸哈希。
- 浏览器会话 Cookie 设置 `HttpOnly`、合适的 `SameSite`，在 HTTPS 下设置 `Secure`；所有有副作用的 Cookie 会话请求必须防护 CSRF。
- CORS 默认不允许跨源管理访问；启用时使用精确来源允许列表，不能同时使用任意来源和凭据。
- 登录、令牌验证和高成本管理操作具备速率限制及失败审计，不向客户端区分账号不存在与密码错误。
- 支持部署在反向代理后；仅信任配置的代理地址段提供的转发头，其他请求中的转发头一律忽略。
- 提供本机 CLI 或容器内命令执行的管理员恢复流程，恢复操作使现有管理会话失效。
- 禁止在访问日志中完整记录订阅查询参数和发布令牌。

#### SEC-002 密钥与敏感信息（P0）

- 订阅 URL、请求头、节点密码和私钥按敏感字段处理。
- 界面、日志、错误报告和诊断包默认脱敏。
- P0 使用应用层认证加密保护数据库中的敏感字段、原始文档和构建产物；仅依赖 SQLite 文件权限不满足要求。
- 正式部署使用独立 `master.key` 文件，不写入数据库。Compose 提供一次性 `init-key` 服务或等价 CLI，在数据卷外生成至少 256 bit 随机密钥，属主设为固定运行身份 UID/GID `65532:65532`、权限设为 `0600`、拒绝覆盖已有文件，运行服务以该非 root 身份只读挂载。
- 环境变量传递主密钥仅允许开发/测试模式使用并显示降低保障警告；生产示例不得采用环境变量秘密。
- 主密钥缺失、权限过宽、格式错误或不可解密数据时 `/readyz` 失败，管理和发布接口不启动；`/healthz` 仅报告不含秘密的初始化错误。
- 密钥派生、加密算法和密文格式版本化，支持不中断服务的密钥轮换。
- 主密钥轮换必须先为全部数据密钥建立旧、新主密钥双 wrapping，再原子替换 secret 目录中的活动文件并切换数据库活动 Key ID；崩溃发生在任一步时，启动流程都必须能按文件 Key ID 和 `prepared` 状态继续或回滚，不能出现只有数据库或只有文件已切换的不可恢复窗口。
- 临时文件、诊断包、异常堆栈和外部校验器输入遵循同一敏感数据策略。
- 含秘密备份必须使用独立备份口令或接收方公钥加密；脱敏备份不得包含可恢复的节点凭据和令牌。
- 应用内“托管备份”是自包含的逻辑备份：敏感数据从应用密文流式解密后立即用独立备份 Key 流式重加密，不包含可直接挂载的 `master.key`，可在新实例中用备份口令或接收方私钥恢复并由目标实例重新加密。直接复制数据目录的冷备份仍必须与 `master.key` 分开保存且两者缺一不可。
- 主密钥缺失或错误时服务进入不可发布的安全失败状态，不得把密文当作空配置覆盖当前结果。

#### SEC-003 网络边界（P0）

- 远程订阅和模板抓取属于服务端请求，必须限制协议、重定向次数、响应大小和超时。
- 默认拒绝 `file://` 等非 HTTP 协议。
- 每次重定向都重新执行协议、端口、主机和目标 IP 策略；连接使用经过校验的解析结果，防止 DNS rebinding 绕过。
- 压缩响应按解压后大小限制；JSON/YAML/Base64 解析限制深度、别名展开、单值长度和总节点数，禁用 YAML 自定义标签及对象实例化。
- 结构化 JSON 输入出现重复对象键时必须以稳定错误拒绝该节点或文档，不得依赖解析库“最后一个值获胜”的不确定行为；来源有效性门槛决定单节点错误是否使整个快照失效。
- 订阅、模板和节点服务器访问环回、链路本地及私网地址默认分别拒绝，可对单个资源显式开启；导入 `10.x` 等 NAS 地址时界面提示授权及风险。
- 探测目标应支持允许列表，避免健康检查被滥用为任意网络扫描器。
- 健康执行器不得开启可从容器外访问的临时代理端口；需要监听时仅绑定隔离的本机地址并使用随机端口和强制生命周期。

#### SEC-004 审计记录（P0）

- 记录登录、认证失败、秘密查看、令牌轮换、来源/模板/流水线/发布目标变更、手工刷新、手工健康检查、回滚、备份和恢复。
- 审计项包含 UTC 时间、操作者、动作、资源、结果和关联 ID；变更内容使用脱敏差异，不记录秘密值。
- 审计记录具有独立保留策略，普通运行日志清理不能一并删除；管理员删除审计记录本身也必须留下事件。

#### OPS-003 可观测性（P0/P1）

- 输出结构化日志并带请求、刷新、检查和构建关联 ID。
- 提供 `/healthz` 和 `/readyz`，不得包含秘密信息。
- P1 提供基础 Prometheus 指标，包括刷新结果、构建结果、检查队列和节点状态计数。
- 日志、快照、健康记录、审计记录和构建产物都有可配置保留期限，并在界面展示预计占用与最近清理结果。默认原始健康记录保留 30 天、小时聚合保留 180 天、运行日志保留 14 天、安全审计保留 180 天。

#### OPS-004 持久化任务调度（P0）

- 刷新、健康检查、构建和清理计划持久化保存；进程崩溃或容器重启后恢复未完成和错过的任务。
- 任务使用租约和幂等处理，崩溃恢复不得并行执行同一资源的两个有效任务。
- 所有时间在内部按 UTC 保存，界面按用户时区显示；周期任务明确记录采用的时区和夏令时处理规则。
- 任务队列提供全局和分类并发上限、排队长度、超时、重试上限和死信/终止状态。
- 关停时停止接收新管理任务，在宽限期内完成或安全释放正在运行的任务。

#### OPS-005 GitHub 与发布供应链（P1）

- GitHub 仓库包含 README、LICENSE、SECURITY、CONTRIBUTING、变更日志和漏洞报告方式。
- CI 执行格式、静态检查、单元测试、集成测试、前端测试、依赖漏洞扫描和多架构镜像构建。
- Release 发布版本化二进制、容器镜像、校验和、SBOM、第三方许可证清单及升级说明。
- 集成 sing-box/Mihomo 执行器的发布物包含其精确版本、许可证、修改状态和对应源码获取方式；CI 对依赖许可证及镜像内容生成可审计清单。
- 容器镜像和发布清单使用项目控制的密钥或无密钥工作流签名，并在文档中提供验证命令。
- 发布流程不得把真实订阅、节点凭据或生产快照作为测试 fixture 或构建上下文上传。

## 6. 格式与协议优先级

### 6.1 输入格式

| 优先级 | 格式 |
| --- | --- |
| P0 | sing-box 完整 JSON、`outbounds` 数组、单节点 JSON；标准 URI 列表和常见 Base64 订阅。 |
| P1 | Mihomo/Clash.Meta 完整 YAML/JSON、`proxies` 列表和 proxy-provider；JSONC。YAML 的键顺序、注释、锚点和排版不属于结构化无损承诺，整份逐字节直通除外。 |
| P2 | Surge、Loon、Quantumult X、Shadowrocket 等客户端格式。 |

### 6.2 节点协议

协议范围不再按 P0/P1/P2 永久分层，所有公开、可合法实现且仍有实际使用的市面协议都属于产品覆盖范围。优先级只决定适配顺序，不决定最终是否支持。

| 覆盖组 | 协议范围 | 交付要求 |
| --- | --- | --- |
| sing-box 生态 | SOCKS、HTTP、Shadowsocks、VMess、Trojan、WireGuard、Hysteria、VLESS、ShadowTLS、TUIC、Hysteria 2、AnyTLS、Snell、Tor、SSH、NaiveProxy，以及稳定版后续新增的代理出站协议。 | 原生对象首先全部无损直通，再逐项补齐标准化、编辑、健康检查与版本校验。 |
| Mihomo/Clash.Meta 生态 | 当前稳定版支持的全部 proxy/provider 协议，包括与 sing-box 重叠的协议以及 SSR、mieru 等扩展。 | 原生 YAML/JSON 无损保留，按目标能力完成双向适配。 |
| 分享链接生态 | SIP002 Shadowsocks、VMess、VLESS、Trojan、Hysteria/Hysteria 2、TUIC、AnyTLS、SOCKS、HTTP 等公开 URI 及主流兼容变体。 | 宽容读取、规范输出，非标准变体需标注来源和兼容行为。 |
| 新协议与私有扩展 | 结构化输入中尚未登记的 `type`、未知字段和客户端扩展。 | 不阻断整份订阅；同格式无损直通并进入待适配清单。 |

协议兼容清单必须对“识别、无损直通、标准化解析、编辑与转换、真实健康检查、目标客户端校验”六项能力分别给出状态。

### 6.3 输出格式

| 优先级 | 格式 |
| --- | --- |
| P0 | sing-box 节点列表与完整配置；首个组合目标为 Momo v1.2.1 + sing-box v1.12.25，另提供独立上游 sing-box v1.12.25 目标。 |
| P1 | Mihomo/Clash.Meta proxy-provider、节点列表与完整配置。 |
| P2 | URI 列表及其他客户端格式。 |

## 7. 默认行为

以下默认值用于减少首次使用时的配置量，均应允许覆盖：

- 同格式输出优先走无损路径；跨格式才走标准化渲染器。
- 新订阅自动刷新周期为 6 小时，请求失败默认重试 5 次，依次等待约 1、3、10、30、60 分钟并加入抖动；`401/403` 使用更慢退避，`404/410` 不立即重试，`Retry-After` 在 6 小时上限内优先采用。
- 订阅失败时使用最后有效快照，不立即删除节点；快照超过 72 小时标记陈旧。
- 节点常规检查周期为 30 分钟，首次失败后分别在 1 分钟和 5 分钟快速复核，单次超时 10 秒，检查结果超过 1 小时标记陈旧。
- 节点连续失败 3 次标记不健康，连续成功 2 次恢复。
- 健康感知输出排除 `unhealthy`，保留 `unchecked` 和执行器暂不支持的 `unsupported`；发布目标可显式改为排除 `unsupported`。
- 不冲突的节点名称保持不变；所有重名节点均保留并依次追加稳定的 ` #2`、` #3` 数字后缀，缺席分配保留 30 天且不自动递补。
- 指纹相同的节点只标记为重复，不自动合并或删除；仅 Health Key 完全相同时复用健康结果。
- 构建结果少于 1 个代理节点，或自动构建相对上一次有效构建骤减超过 50% 时暂停发布；首次构建只应用绝对门槛。
- 所有输出使用最后有效构建，生成和切换采用原子操作。

## 8. 非功能需求

### 8.1 正确性

- 每个解析器和渲染器必须有正常样例、边界样例和错误样例。
- 无损路径使用 golden test，覆盖未知字段、`null`、嵌套对象、数组和新增协议类型。
- 每种修改操作必须证明只改变声明的字段路径。
- 使用真实 sing-box/Mihomo 二进制做版本化集成测试，不仅验证 JSON/YAML 可解析。

### 8.2 可靠性

- 进程重启不得丢失已确认的源快照、健康状态和当前发布版本。
- 刷新、健康检查和构建任务可重试且具有幂等性。
- 单个坏订阅、坏节点或坏模板不能阻塞其他任务。
- 外部网络不可用时，已生成的订阅仍能正常读取。

### 8.3 性能基线

首个正式版以单实例管理 100 个订阅源、20,000 个节点为设计上限：

- 在 2 vCPU、2 GiB 内存、SSD、本地 Docker 的 `linux/amd64` 或 `linux/arm64` 参考设备上，以 20 个并发连接、产物不超过 5 MiB 且已预热条件读取已构建订阅，服务端处理时间 P95 小于 100 ms，不含客户端网络时间。
- 在同一参考设备上，10,000 个节点的纯内存流水线 P95 在 2 秒内完成，不含远程请求和真实健康检查。
- 健康检查必须受并发和速率限制，不能为追求速度耗尽文件描述符或带宽。
- Web 界面的大列表使用分页或虚拟滚动。
- 性能报告记录 CPU、内存、磁盘、服务版本、数据规模和命令，发布版本之间可比较。

### 8.4 兼容性

- 数据库、内部节点模型、REST API 和插件接口分别版本化。
- 输出适配器声明支持的客户端版本范围。
- 客户端新增未知字段时，无损路径无需等待本项目升级即可保留该字段。

### 8.5 可维护性

- 解析、标准化、操作、渲染、健康检查和发布模块边界清晰。
- 核心领域逻辑不依赖 HTTP 框架和界面。
- 对外 API 从 `/api/v1` 起步，并生成 OpenAPI 文档。
- 数据迁移、备份恢复和目标版本升级必须有自动化测试。

### 8.6 安全质量门槛

- 发布流水线覆盖认证绕过、CSRF、CORS、SSRF/重定向、DNS rebinding、解析资源耗尽、路径穿越和秘密脱敏测试。
- 依赖扫描发现已知 Critical 漏洞时阻止发布；High 漏洞需要有记录的影响判断、缓解措施和修复期限。
- 使用固定脱敏测试凭据验证日志、指标、审计、错误响应、诊断包和外部执行器输出均不包含秘密。
- 备份恢复、主密钥轮换和发布令牌吊销通过端到端测试。

### 8.7 易用性质量门槛

- P0 核心工作流至少具备桌面 1440x900、平板 768x1024 和手机 390x844 的自动化端到端测试与截图检查。
- 测试覆盖首次启动、导入来源、解决重名、创建输出、查看健康故障、回滚和恢复错误状态。
- 动态节点名称、长 URL、长协议名和本地化文本不能遮挡操作或撑破固定控件。
- 键盘与可访问性自动检查作为 P1 发布门槛，并保留人工抽查清单。

## 9. 关键验收场景

### AC-001 sing-box 未知字段无损

给一个 sing-box 节点增加当前系统未知的嵌套字段，导入后选择 sing-box 节点输出且不应用操作。将输入和输出节点解析为 JSON 后应深度相等，构建无降级警告。

### AC-002 只改节点名称

导入带 ECH、uTLS、Reality 或其他扩展字段的 sing-box 节点，触发名称冲突。输出节点除 `tag` 外应与原对象深度相等，并在构建差异中仅出现 `/tag`。

### AC-003 同网站多订阅稳定命名

从相同主机名导入两条订阅，两者包含名称相同以及指纹完全相同的可用节点。系统保留全部节点，并输出 `香港 01`、`香港 01 #2`、`香港 01 #3` 三个唯一名称；交换源顺序、刷新内容和重启容器后，名称集合保持一致。

### AC-004 订阅临时故障

一个已成功的订阅连续两次返回 500。当前发布结果继续使用最后有效快照，来源显示降级并记录失败；不得发布空结果。

### AC-005 节点摘除与恢复

节点连续三次真实代理检查失败后，从启用健康过滤的输出中消失；连续两次成功后重新加入。两次变化均产生清晰构建记录，其他节点名称和顺序不抖动。

### AC-006 探测目标整体故障

探测站点异常导致多数节点在短时间同时失败。系统识别为检查基础设施异常，暂停批量摘除，继续提供最后有效构建并显示告警。

### AC-007 坏配置不发布

新模板引用不存在的出站，或目标配置验证器失败。构建状态为失败，固定订阅 URL 仍返回上一份有效结果；验证清单说明本次使用了内部规则还是目标客户端/内核检查。

### AC-008 有损转换可见

将含目标格式无法表达字段的节点跨格式输出。预览明确列出字段和受影响节点；未获得该发布目标的明确允许时构建失败。

### AC-009 新协议不中断服务

导入一个含系统未知 `type` 和未知嵌套字段的 sing-box 节点。系统能够保存、展示来源并在 sing-box 输出中深度相等地无损直通；能力矩阵将其标记为“可直通、待解析”，跨格式转换不得伪造成功。

### AC-010 完整配置用途明确

导入同时含代理节点、逻辑出站、DNS、路由和入站的完整 Momo 配置。分别选择节点源、模板、模板与节点源、整份直通后，预览和生成结果应符合 SRC-006；切换用途创建新修订，已有构建清单仍引用旧修订。

### AC-011 构建可复现且不被旧任务覆盖

固定构建清单连续重放两次应得到逐字节相同产物。构建运行期间发布新流水线修订并完成新构建后，即使旧任务最后才结束，也不能把发布端点切回旧输入结果。

### AC-012 敏感数据安全失败

使用固定测试秘密完成导入、刷新、构建、健康检查、失败诊断、备份和令牌轮换。日志、API 错误、指标、审计差异和临时文件中不得出现秘密；使用错误主密钥重启时服务不发布空或损坏结果。

### AC-013 不可信输入与网络边界

分别提交超深 JSON、YAML 别名膨胀、超大压缩响应、跳转到私网的 URL 和 DNS rebinding 测试源。请求应在配置资源上限内失败并给出稳定错误码，进程保持可用，已有发布端点继续返回最后有效构建。

### AC-014 任务崩溃恢复

在刷新、检查和构建任务执行中强制终止容器并重启。任务应按租约恢复或明确终止，不产生两个并发有效任务，不重复切换发布版本，并保留可追踪的任务记录。

### AC-015 数字后缀生命周期

依次导入三个同名节点，确认获得原名、` #2`、` #3`；删除持有原名的节点后，其余两个不得改名。随后新增第四个同名节点、改变来源顺序并重启容器，已有分配保持不变；原节点在 30 天保留期内恢复时取回原名。手工压缩编号必须生成可预览的新修订。

### AC-016 健康结果安全复用

两个原始节点指纹相同，其中一个经输出流水线修改 `server_name` 或传输字段。两者 Health Key 必须不同并执行两次探测；另一个未修改副本 Health Key 相同，只执行一次实际探测但生成两条独立节点关联记录。

### AC-017 密钥初始化与只读运行

执行 `init-key` 生成数据卷外 `0600` 主密钥，再次执行不得覆盖。缺失、权限过宽和错误密钥分别使 `/readyz` 失败且不开放管理/发布接口；挂载正确密钥、只读根文件系统和受限 tmpfs 后，刷新、构建和健康执行器正常工作且不留下秘密临时文件。

### AC-018 健康队列过载

在参考设备上调度 20,000 个唯一 Health Key，并持续插入重复周期任务和手工复测。队列不得超过 100,000 个唯一 Health Key，重复任务被合并，手工复测在 2 个预留槽位内优先开始；排队超期的节点标记陈旧，服务读取现有发布产物的 P95 仍满足性能门槛。

### AC-019 健康过滤不改变名称分配

同时导入两个同名节点，其中稳定排序在前的节点首次检查即为 `unhealthy`，另一个为 `healthy`。启用健康过滤时只输出后者但仍保持其既有后缀；前者恢复后取回原分配，两个节点和其他冲突节点均不得因健康状态变化重新编号。

### AC-020 主密钥轮换崩溃恢复

分别在新 Key 文件生成后、双 wrapping 提交后、`master.key` 原子替换后和数据库活动 Key ID 切换后强制终止进程。每个故障点重启都必须使用当前文件 Key ID 自动继续或安全回滚，所有历史密文与 HMAC 身份保持可读且服务不会用空数据启动。

### AC-021 控制探测、批次与全局队列

让两个 Output 在同一窗口引用部分相同 Health Key。数据库只存在一个活动物理队列项和一次实际节点探测，两个 Batch Item 关联同一 Health Record；直连控制使用独立 Control Record。执行器崩溃和目标错误不进入 10/50% 有效样本分母，节点归因失败按各 Output 批次独立熔断。

### AC-022 托管备份与跨实例恢复

创建含固定测试秘密的口令加密托管备份，在全新实例只使用备份文件和口令完成预检与恢复，不提供原 `master.key`。恢复后的数据使用目标实例 Key 重新加密，备份、暂存和日志无明文秘密；错误口令、损坏流和过期预检均不改变当前实例。

### AC-023 重复 JSON 键与无损数字

导入包含重复对象键、超出 IEEE-754 安全整数范围、指数形式和尾随小数零的 JSON fixture。重复键按策略稳定拒绝且不采用任意值；其余数字在未修改输出中保留原词法，不透明指纹不经过浮点转换，并在重放时保持一致。
